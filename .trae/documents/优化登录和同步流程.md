# Goalify 登录和同步流程优化计划

## 需求概述

1. 打开应用后直接进入项目页面（而非登录页面）
2. 支持离线使用：不登录也能创建项目、任务、子任务（owner_id 为空）
3. 登录后自动同步：填充无 owner_id 的记录，并同步到云端
4. Header 区域显示同步状态标识
5. 登录后定时同步（高性能方案）

### 步骤 5: 实现高性能定时同步方案

**策略**: 智能增量同步 + 多种触发机制

#### 5.1 同步触发时机

| 触发方式 | 时机 | 说明 |
|---------|------|------|
| **登录时** | 用户登录成功后 | 立即执行一次完整同步（push + pull） |
| **手动同步** | 用户点击同步按钮 | 立即执行同步 |
| **定时同步** | 每 5 分钟 | 仅当数据有变化时执行 |
| **应用恢复** | 从后台切换到前台 | 检查是否需要同步 |
| **数据变更** | 创建/更新/删除后 | 延迟 3 秒执行增量同步 |

#### 5.2 性能优化策略

1. **变更追踪（Change Tracking）**
   - 使用 SQLite 的 `updated_at` 字段
   - 记录上次同步时间戳
   - 只同步 `updated_at > last_sync_time` 的记录

2. **批量操作**
   - 使用 Supabase 的批量插入/更新 API
   - 减少 HTTP 请求次数

3. **防抖处理**
   - 数据频繁变更时，合并多次同步请求
   - 使用 3 秒防抖延迟

4. **网络状态检测**
   - 检测网络连接状态
   - 离线时暂停同步，恢复后自动执行

#### 5.3 前端实现

**文件**: `packages/web/src/stores/AuthStore.ts` 或新建 `packages/web/src/stores/SyncStore.ts`

**实现内容**:
```typescript
// 同步状态管理
interface SyncState {
  status: 'idle' | 'syncing' | 'error' | 'offline';
  lastSyncedAt: number | null;
  pendingChanges: number;
  
  // 动作
  sync: () => Promise<void>;
  startPeriodicSync: () => void;
  stopPeriodicSync: () => void;
}

// 定时同步逻辑
- 使用 setInterval，间隔 5 分钟
- 检查是否有未同步的变更（通过 sync_queue 表或时间戳）
- 仅在需要时执行同步
- 页面不可见时暂停定时器（使用 Page Visibility API）
```

### 步骤 6: 数据库变更追踪增强

### 步骤 7: 错误处理和重试机制

- 点击后弹出登录对话框（不跳转页面）
- 登录成功后保持当前页面，自动开始同步

## 性能考虑

1. **首次同步优化**
   - 数据量大时，使用分页同步
   - 显示同步进度

2. **内存优化**
   - 同步完成后释放大对象引用
   - 避免同时执行多个同步任务

3. **电池优化**
   - 后台时降低同步频率
   - 使用 requestIdleCallback（Web）或类似机制
